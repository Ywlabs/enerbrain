# EnerBrain — RAG·LLM 연계 참고 (연구 메모 반영)

과거 별도 연구 노트에서 정리한 내용을 EnerBrain 기준으로 옮긴 문서다. **상황 판단**, **자연어 설명·요약**, **운영 규정·매뉴얼 질의** 등에 LLM을 붙일 때 아래를 기본 가정으로 삼을 수 있다.

---

## 1. 벡터 저장소(컬렉션) 전략

- **권장**: **단일 통합 컬렉션(또는 논리적으로 하나의 인덱스)** + **메타데이터 필터** (`source_type`, `site_id`, `equipment_id`, `doc_version` 등).
- **이유**: 설비·문서 유형이 늘어도 검색·운영 복잡도를 줄이고, 복합 질의에 한 번의 검색으로 대응하기 쉽다.
- **예외**: 테넌트·고객사 **물리 격리**가 필수일 때만 컬렉션/인스턴스 분리 검토.

구현체(Chroma, pgvector, Qdrant 등)는 나중에 고르고, **패턴은 위와 동일**하게 가져가면 된다.

---

## 2. 원천 데이터·동기화 (Postgres와 정합)

- 파일 폴더만 주기적으로 스캔하는 방식보다, **지식 원천 메타를 DB에서 관리**하는 편이 EnerBrain(Postgres 예정)과 잘 맞는다.
- 연구안의 **`rag_metadata` 유사 테이블** 개념 적용:
  - 문서 식별, 저장 경로/URI, 제목, 버전, 분류, **`ACTIVE` / `INACTIVE`** 로 RAG 포함 여부 제어.
  - 임베딩·벡터 동기화 작업(배치/스케줄러)은 **이 테이블을 단일 진실 공급원**으로 삼아 예측 가능한 파이프라인 구성.

---

## 3. 피드백 루프(품질 개선)

- 챗·설명 UI에서 **좋음/나쁨** 등 평가를 받을 경우:
  - 질문, 검색된 컨텍스트, 최종 답변, 평가를 **`feedback_logs` 유형**으로 저장.
  - 나쁜 평가는 **검색 실패**(문서·청크 보강) vs **생성 실패**(프롬프트·모델 설정)로 나눠 분석.
- EnerBrain은 **에너지 도메인**이므로, 잘못된 답이 **운영 안전**과 연결될 수 있어 로그·감사 추적 가치가 크다.

---

## 4. 모듈 배치와의 관계 (baseline 모듈 표)

| baseline 모듈 | RAG·LLM과의 관계 |
|---------------|------------------|
| Inference API | RAG 보강 질의·요약 결과를 JSON으로 서빙하는 엔드포인트로 확장 가능 |
| Data Analyzer | RAG용 전처리·청킹·메타 추출과 겹치는 부분은 파이프라인만 분리해 중복 방지 |
| 로컬 LLM / 외부 API | `rag_extend`의 생성 단계; 추론 코드는 `app/services/llm` 등으로 격리 |

---

## 5. 정리

- RAG는 EnerBrain **필수 1차 범위**라기보다 **확장 시 설계 참고**로 두되, Postgres·배치·감사 로그와 **같은 철학**으로 맞추면 이후 도입 비용이 줄어든다.
- 상세 알고리즘·라이브러리 선정은 스택 확정 후 `techContext`·별도 스펙에 갱신한다.
