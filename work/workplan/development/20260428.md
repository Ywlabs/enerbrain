# 2026-04-28 개발 계획 (분석 공통 엔진 1차)

## 1) 목표

- 분석항목 설정(`TB_ANALYSIS_ITEM`)을 기반으로 분석모듈을 공통 방식으로 실행하는 엔진을 구현한다.
- 실행 결과를 표준 포맷으로 `TB_ANALYSIS_RUN`에 기록하여, 운영 API(`analysis-runs`)에서 즉시 조회 가능하게 만든다.
- 개별 알고리즘 코드와 실행 인프라(로딩/재시도/타임아웃/로깅/결과저장)를 분리해 유지보수성을 확보한다.

---

## 2) 범위 (이번 단계 포함)

## 포함

- `module_runner` 실행 엔진 1차 구현
- 분석 실행 컨텍스트/결과 표준 스키마 정의
- 동적 모듈 로딩(`MODULE_PATH_CN`, `ENTRY_FUNC_NM`) 구현
- 실행 상태 관리(RUN/DONE/FAIL) 및 처리시간 계산
- 재시도(`RETRY_CNT`) 및 타임아웃(`TIMEOUT_SEC`) 처리
- 실행 결과 DB 적재(`TB_ANALYSIS_RUN`)
- 샘플 분석모듈 1개 연결(E2E 검증)

## 제외

- 다중 워커/분산 실행
- 큐 기반 비동기 파이프라인
- 고급 모니터링(알림, 메트릭 대시보드)
- 고가용성 스케줄러(클러스터 락 등)

---

## 3) 아키텍처 설계

## 3.1 공통 실행 흐름

1. 실행 대상 분석항목 로드  
2. `TB_ANALYSIS_RUN`에 `RUN` 상태로 시작 레코드 생성  
3. 모듈 로딩 및 엔트리 함수 호출  
4. 성공 시 `DONE` + 결과 JSON 저장  
5. 실패 시 재시도 후 `FAIL` + 에러 저장  
6. 실행 종료 시 처리시간/입출력 건수/종료시각 업데이트  

## 3.2 책임 분리

- `runner/module_runner.py`
  - 실행 오케스트레이션(로딩/호출/재시도/타임아웃/예외처리)
- `runner/result_writer.py`
  - `TB_ANALYSIS_RUN` 쓰기 전담
- `domain/analysis/repository.py`
  - 실행 대상 조회/실행이력 조회 쿼리
- `runner/scheduler.py`
  - 주기 실행 트리거(이번 단계는 최소 기능)

---

## 4) 표준 인터페이스 정의

## 4.1 RunContext (엔진 -> 모듈 입력)

- `analysis_item_no`
- `biz_no`
- `site_no`
- `params` (`PARAMS_JSON`)
- `run_id` (`ANALYSIS_RUN_NO`)
- `started_at`

## 4.2 RunResult (모듈 -> 엔진 출력)

- `status` (`DONE`/`FAIL`)  
- `input_nocs` (입력 건수)
- `output_nocs` (출력 건수)
- `result_json` (`RUN_RSLT_JSON`)
- `model_json` (`RUN_MODEL_JSON`)
- `error_message` (실패 시)

## 4.3 모듈 함수 시그니처(표준)

- `def run(context: dict) -> dict`
- 반환 dict를 엔진에서 `RunResult` 표준 키로 매핑

---

## 5) 데이터/상태 정책

## 5.1 상태 전이

- 시작: `RUN`
- 성공: `DONE`
- 실패(최종): `FAIL`

## 5.2 에러 정책

- 예외 발생 시 스택 핵심 메시지를 `ERR_CN`에 저장
- 재시도 중 실패 이력은 로그로 남기고 최종 실패만 DB 확정

## 5.3 JSON 정책

- `RUN_RSLT_JSON`: 실행 요약(정확도, 통계, 결과 개요)
- `RUN_MODEL_JSON`: 모델 경로/버전/메타/지표
- 키 네이밍은 snake_case 고정

---

## 6) 세부 구현 작업 (WBS)

## WBS-01 실행 대상 로딩

- 분석항목 조회 함수 정리
- 비활성/삭제(`USE_YN`, `DEL_YN`) 제외 정책 확정

## WBS-02 run_id 생성 및 시작 기록

- `ANALYSIS_RUN_NO` 생성 규칙 정의
- `RUN_BGNG_DT`, `RUN_STTS_CD='RUN'` 초기 저장

## WBS-03 동적 모듈 로딩

- 파일 경로 유효성 검증
- 엔트리 함수 존재 여부 검증
- 보안상 허용 루트 경로 제한 검토

## WBS-04 실행 제어

- 타임아웃 처리
- 재시도 루프(`RETRY_CNT`)
- 예외 캡처 및 에러 정규화

## WBS-05 종료 기록

- `RUN_END_DT`, `PROC_MSEC`
- `INPUT_NOCS`, `OUTPUT_NOCS`
- `RUN_RSLT_JSON`, `RUN_MODEL_JSON`
- `ERR_CN` 저장

## WBS-06 샘플 모듈 E2E

- 샘플 모듈 1개 생성
- 분석항목 1건 등록 후 엔진 실행
- `analysis-runs` API에서 결과 확인

## 8) 완료 기준 (DoD)

- 분석항목 1건을 엔진으로 실행해 `TB_ANALYSIS_RUN`에 결과가 저장된다.
- 실패/재시도/타임아웃 케이스가 테스트로 검증된다.
- `analysis-runs` API에서 실행 결과가 조회된다.
- 기존 PAS 테스트가 모두 통과한다.
- 운영 로그에서 실행 추적이 가능하다(시작/종료/실패).
