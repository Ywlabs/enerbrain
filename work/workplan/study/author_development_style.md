# WorkPlan 작성자 개발·협업 스타일 (AI 참고용)

본 문서는 `work/workplan/`에 축적된 **EnerHub(YWLab) 시절** 기획·지시 문서를 바탕으로, 작성자의 **일 시키는 방식·꼼꼼함·기술적 선호**를 추려 정리한 것이다. **EnerBrain** 작업 시 에이전트가 톤·범위·산출물 형태를 맞출 때 참고한다.

> 원본 출처(분석 후 삭제됨): 예전 `work/workplan/202507/`, `202606/` 내 `20150701*`·`20150701_2_*`·`20250622.md`·`notification_usage_examples.md` 등. 요약만 본 문서에 남김.

---

## 1. 지시·명령 톤

- **한국어** 위주, 필요 시 영어 기술 용어·경로 혼용.
- **직설적 지시**: 「~해주기 바람」「~하면 된다」「~말아라」「무조건 유지」 등 **요구를 문장으로 고정**하는 편.
- **범위 통제가 강함**: 「개선과 **관계 없는 부분은 절대 수정하지 말 것**」「기존 **route / service 패턴·파일명 유지**」처럼 **손대면 안 되는 경계**를 명시.
- **승인 게이트**: 「**설계부터 하고 컨펌 받은 뒤** 구현」「먼저 파악·제안 후 진행」— 큰 변경은 **문서/계획 → 사용자 확인 → 코딩** 순서를 기대하는 경우가 많음.
- **맥락 공유**: 실제 작업 시 **절대 경로**(예: `D:\Develop\workspace-ywlabs\...`)로 파일을 짚어 주거나, 「현재 동작하는 흐름」을 단계 번호로 나열.

에이전트는 위를 **존중**하고, 불명확하면 짧게 되물은 것이 안전하다.

---

## 2. 꼼꼼함·산출물 깊이

작성자 문서에는 다음 층이 **반복적으로** 나타난다.

| 층 | 내용 예시 |
|----|-----------|
| **목적·배경** | 개선 목적, 현재 동작 요약, 이미 완료된 부분은 「최대한 유지」 |
| **기능 스펙** | 폼 필드별(유형, readonly, validation, 팝업, 토스트) **항목 단위** 나열 |
| **API·데이터** | HTTP 메서드·경로, Request/Response JSON 예시, 테이블·SQL 스크립트 요청 |
| **프론트** | 레이아웃·컴포넌트 수준 설계, **참고할 기존 화면** 명시(예: 다른 Vue 파일과 동일 UX) |
| **비기능** | 보안(XSS, SQLi, CSRF), 성능(응답 시간, 부하), 접근성(WCAG), 에러 시나리오 |
| **운영** | 배포 절차, 로그 경로, 마이그레이션·백업 명령 |
| **체크리스트** | 구현 단계별 `- [ ]`, 품질·배포·문서화까지 **빠짐없이** |

즉, **기능만이 아니라 검증·배포·문서까지 한 번에 요구**하는 스타일에 가깝다.

---

## 3. 기술·구조에 대한 반복 선호

- **백엔드**: **라우트 + 서비스** 분리, 응답은 **`ApiResponse` 형식**(success / message / data) 통일.
- **프론트**: Vue 기반 시 **기존 패턴·파일 구조 존중**, 특정 화면을 **템플릿처럼 복제**해 달라는 지시.
- **비동기·백그라운드**: 장시간 작업은 **큐·스케줄러** 등으로 분리하고, UI 이탈 후에도 **상태 조회·다운로드** 가능하게 설계하길 원함.
- **리스크 명시**: 기술 검토 요청 시 **이슈 후보를 번호로 나열**(예: 브라우저 종료 시 작업 끊김, 성능)하고, 연관 파일 **전체 점검**을 요구.

EnerBrain은 스택이 FastAPI 등으로 바뀌었으므로 **원칙(패턴·ApiResponse·범위 최소화·설계 선행)** 은 따르고, **Flask/Vue 경로 문자열**은 저장소 기준으로 치환해 해석하면 된다.

## 4. 문서 형식 습관

- Markdown 제목에 **이모지·굵게**로 섹션 구분(📋 구현 계획, 🔒 보안 등).
- 긴 글은 **목차·단계(1단계, 2단계)**.
- **코드 블록**으로 명령어·설정·Vue 스니펫·Python 예시를 그대로 적어 두어 **복붙·대조** 가능하게 함.
- `baseline.md` 형태: 한 줄 정의, 표, 「해줘야 할 일」로 **다음 액션**을 짧게 닫음.

---

## 5. 에이전트가 맞추면 좋은 행동

1. **범위**: 요청 범위 밖 리팩터·스타일 정리 **자제**; 필요하면 먼저 제안.
2. **산출**: 가능하면 **설계 요약 → 사용자 확인** 후 구현; 큰 기능은 **체크리스트**로 남겨도 됨.
3. **응답 형식**: API는 **통일된 JSON 계약**; 프론트는 **validation·에러 메시지·토스트**까지 스펙에 맞출 것.
4. **검증**: 보안·성능·접근성을 **스스로 한 번 훑고** 빠진 부분을 짚어 주면 작성자 기대와 잘 맞음.
5. **문서**: WorkPlan에 맞춰 **파일 경로·엔드포인트·마이그레이션**을 구체적으로 적어 두면 이후 동기화에 유리.

## 6. EnerBrain에서의 주의점

- 본 workplan은 **타 프로젝트(EnerHub) 경로·Flask**를 포함한다. **현재 저장소의 `memory-bank/`, `.cursor/rules/`, `backend/app/`** 가 사실상 기준이다.
- 「참고만 하고 신규로 만든다」는 `baseline.md` 원칙과 같이, **과거 문서와 현재 코드가 충돌하면 EnerBrain 쪽을 우선**한다.
