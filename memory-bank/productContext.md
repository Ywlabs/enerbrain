# EnerBrain — 제품 맥락

## 왜 존재하는가

에너지 관련 **다양한 소스(DB 로우, 센서, 이력)** 를 한곳에서 **해석·예측·알림**까지 연결하는 **분석 허브(두뇌)** 가 필요하다. 단순 조회가 아니라 **통계·AI·상황 판단**이 포함된다.

## 해결하려는 문제

- 원시 데이터와 운영 의사결정 사이의 **전처리·집계·품질(결측 등)** 격차.
- **예측**(수요·공급 패턴)과 **이상·고장 징후**에 대한 **자동화된 신호**.
- **실시간** 흐름(MQTT 등)과 **배치** 분석의 공존.

## 기대 동작(개념)

1. 외부 시스템(예: enerhub) 또는 내부 배치가 **분석·추론을 요청**한다.
2. **Data Analyzer**가 DB/스토리지에서 데이터를 읽어 정제·피처를 만든다.
3. **Forecaster / Fault Detector / LLM·정책 모델** 등이 단계적으로 붙어 **결과를 산출**한다.
4. **Inference API**가 **일관된 JSON 계약**으로 응답한다.
5. 필요 시 **MQTT**로 실시간 반영 또는 명령 전달.

## 사용자·운영 경험 목표

- API 응답 형식은 **명확한 성공/메시지/데이터** 구조(기존 WorkPlan `ApiResponse` 패턴)로 통일.
- OpenAPI(` /docs`)로 **계약을 문서화**해 프론트·외부 연동 비용을 줄인다.
- 환경별 설정은 **`.env` + Pydantic Settings**로 분리해 배포를 단순화한다.

## 프론트엔드(대시보드) 방향

- **목적**: **AI 분석 현황**, **자원**, **시간 축**, **이력** 등을 한눈에 보고 관리하는 **운영 대시보드**.
- **기술 선택은 미정**: **Vue 3 기반 커스텀** vs **Grafana 등 솔루션** 위주 — 추후 결정. 상세 가이드는 `.cursor/rules/frontend-rules.mdc` 참고.
